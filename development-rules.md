# 開発ルール

本プロジェクトは **オブジェクト指向プログラミング (OOP)** を基本とし、以下の原則に従って開発を行う。

---

## 0. Clean Architecture の採用

本プロジェクトは **Clean Architecture** を設計方針として採用する。

### 基本方針
- **関心の分離**: ビジネスロジックと外部 IO (API・DB・ファイル等) を明確に分離する。
- **依存性の方向を統一**: 依存は常に「外側 → 内側」の一方向。内側 (ドメイン / ユースケース) は外側 (インフラ) を知らない。
- **抽象に依存する**: 上位層は具象ではなく抽象 (インターフェース) に依存する (DIP)。

---

## 1. KISS 原則 (Keep It Simple, Stupid)

「できるだけシンプルに保つ」という設計思想。

### 基本方針
- 複雑な実装より、単純で理解しやすい実装を優先する。
- 過剰な抽象化・汎用化を避ける (YAGNI と併用)。
- コードは「将来誰が読んでも理解できるか」を基準に書く。

### 実践ルール
- 1 つの関数・メソッドは 1 つのことだけを行う。
- ネストは浅く保つ (目安: 3 段階まで)。
- 不要なデザインパターンを導入しない。必要になってから導入する。
- 「動く最小のコード」をまず書き、必要に応じてリファクタする。
- コメントで補わないと理解できないコードは、コード自体を書き直す。

---

## 2. SOLID 原則

オブジェクト指向設計における 5 つの基本原則。

### S — 単一責任の原則 (Single Responsibility Principle)
> クラスが変更される理由は、ただ 1 つでなければならない。

- 1 つのクラスは 1 つの責務のみを持つ。
- 責務が複数あると感じたら、クラスを分割する。

### O — 開放/閉鎖の原則 (Open/Closed Principle)
> ソフトウェアの構成要素は、拡張に対して開いており、修正に対して閉じていなければならない。

- 既存コードを修正せずに機能追加できる構造にする。
- 継承・インターフェース・ポリモーフィズムを活用する。

### L — リスコフの置換原則 (Liskov Substitution Principle)
> 派生クラスは、基底クラスと置換可能でなければならない。

- サブクラスはスーパークラスの契約 (事前条件・事後条件) を破らない。
- 継承は「is-a」の関係が成立する場合のみ使用する。

### I — インターフェース分離の原則 (Interface Segregation Principle)
> クライアントは、使用しないメソッドへの依存を強制されるべきではない。

- 巨大なインターフェースを作らず、役割ごとに小さく分割する。
- 必要最小限のメソッドだけを公開する。

### D — 依存性逆転の原則 (Dependency Inversion Principle)
> 上位モジュールは下位モジュールに依存してはならない。両者は抽象に依存すべきである。

- 具象クラスではなく、抽象 (インターフェース / 抽象クラス) に依存する。
- DI (依存性注入) を活用し、結合度を下げる。

## 3. AI 駆動開発のガードレール

AI にコードを書かせる際に陥りやすい失敗を防ぐためのルール。

- **推測で書かない**: 存在を確認していない API・仕様を書かない。不明なら読む。
- **依頼範囲を逸脱しない**: 頼まれていないリファクタ・整形をついでに行わない。
- **未使用の保険コードを書かない** (YAGNI): 使われない分岐・将来用の引数を足さない。
- **コメントは WHY のみ**: 何をしているかはコードが語る。なぜその選択かだけ書く。
- **書いたら動かす**: 変更後は最小の import / 実行で通ることを確認してから完了とする。

---

## 4. 型アノテーション (契約の明示)

### 判断
**全関数で型アノテーションを使う**。総合的にメリットがデメリットを上回る。

### メリット
- IDE 補完・静的解析 (mypy / pyright) で誤りを実行前に検出できる (フェイルファストの延長)。
- 関数の契約 (入力・出力の形) がシグネチャで完結し、本文を読まなくても使い方が分かる。
- リファクタ時に型違反が連鎖的に露呈し、変更漏れが減る。
- AI が書く / 読むコードで特に効く。シグネチャから意図を掴めるため、推測ミスが減る。

### デメリットと折衷
- **動的データ (JSON など) は結局 `Any` になりがち**で型安全性は名目的 → それでも `dict[str, Any]` と明示することで「動的データを扱っている」という意図が伝わる。裸の `dict` より優位。
- **維持コストが上がる** → 厳しいルール (mypy --strict / TypedDict / Protocol 強制) は採用しない。最低限「全関数で要素型まで明示」だけを守る。
- **小さな private ヘルパーで冗長** → 引数 1-2 個でも書く。揺れがあると統一感が崩れるため例外を作らない。

### 実践ルール
- **全関数** (public / private / module level / nested いずれも) の引数と返り値に型を付ける。
- **裸の `dict` / `list` / `tuple` / `set` を書かない**。要素型を明示する。動的なら `dict[str, Any]` のように `Any` を使う。
- **`Any` の使用を恐れない**。「動的型を扱っている」ことを明示する目的での使用は推奨。一方、「とりあえず `Any`」で型検査を回避する目的での使用は禁止。
- **型エイリアスは再利用される場合のみ作る**。1 ファイル内でしか使わない型には名前を付けない (KISS)。
- 既存コードに混在を見つけたら、その関数のシグネチャだけ揃える (周辺の大規模リファクタは別タスク)。

### 採用しないもの (KISS との緊張で見送り)
- `mypy --strict` の CI 強制 (導入コストが当面の利益を上回る)。
- `TypedDict` / `Protocol` の強制 (動的 JSON 周りで冗長になる)。
- ジェネリクス / `TypeVar` の積極利用 (必要になった時にだけ書く)。

## 5. フェイルファスト (明示的にエラーを起こす)

バグは握り潰されると発見が遅れる。エラーは隠さず早く出す。

- **フォールバックは原則不要**: 「一応動く」状態を作るためのフォールバックは障害検知を遅らせる負債。必要性が証明されるまで足さない。
- **フォールバックを増やさない**: 新しい try/except や `or default` を足す前に、失敗しない前提が正しいかを疑う。
- **エラー時は明示的に落とす**: 想定外の状態は握り潰さず例外を送出する。早期クラッシュが早期発見に繋がる。
- 例外: 通知・ログ等「副作用の失敗」は本処理を止めないため捕捉して良い。

---

## 6. 常に根本解決を目指す (対症療法禁止)

バグや不具合を直すときは、症状を打ち消す分岐・フォールバック・リトライを足すのではなく、**なぜその症状が出る設計になっているか** に踏み込んで原因から消す。

### 基本方針
- 修正案を出す前に「これは症状を隠しているだけか、原因を消しているか」を必ず自問する。
- データの **出所が正しいか** (どの API / どのオブジェクト / どの責務のレイヤから取っているか) を先に疑う。出所が間違ったままフィルタやソートで辻褄を合わせない。
- 設計レベルの歪みは設計で直す。分岐追加は最後の手段。

### 実践ルール
- 「並び順が不定だからソート」「空のときフォールバック」「たまに失敗するからリトライ」を思いついたら、一度立ち止まって **そもそも正しいソースから取っているか** を確認する。
- 責務の境界が曖昧なまま条件分岐を積まない。必要なら責務を分離する (SRP)。
- 根本修正が現実的でない場合は「これは応急処置で、根本は X」と明示し、根本修正のタスクを残す。
- 対症療法でその場をしのいだまま放置しない。

---

## 適用指針

| 状況 | 優先する原則 |
| --- | --- |
| 新しい機能を追加するとき | KISS → SOLID (O, D) |
| クラスが肥大化してきたとき | SOLID (S, I) |
| 継承を使おうとしているとき | SOLID (L) |
| エラー処理を書こうとしたとき | フェイルファスト優先、フォールバックは足さない |
| バグを修正するとき | 根本解決優先、分岐追加で症状を隠さない |
| 関数を書く / 編集するとき | 型アノテーションを要素型まで明示 (裸 `dict` 禁止) |
| 迷ったとき | **KISS 優先** |

> SOLID はあくまで設計の指針であり、目的ではない。
> 過剰に適用すると KISS に反するため、**バランス**を常に意識する。
